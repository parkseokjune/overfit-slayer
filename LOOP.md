# LOOP.md — 자율 운용/업그레이드 루프 지침 (v2)

> Claude가 `/loop`로 반복 실행할 때 따르는 지침. 구버전(Phase 0~6 개발 루프)은 완료되어
> LOG.md Iteration 1~18에 기록됨. 현재는 **운용 감시 + 자율 업그레이드** 모드.

## 시스템 현황 전제
- 24시간 러너가 백그라운드에서 매매 중 (60초 리스크 틱 / 15분 신호 / 일·주·월 자동 작업)
- 전략·성과·백로그는 STATE.md, 전체 이력은 LOG.md 참조

## 매 이터레이션 절차

1. **상태 파악**: STATE.md(백로그·채택전략)와 `logs/runner.log` 최근 줄 확인
2. **운용 점검** (필수, 1분):
   - 러너 살아있는가? `results/ALERT.txt` 생겼는가? 킬스위치(halted) 발동했는가?
   - 체결 발생 시: 체결가 vs 시그널가 슬리피지를 백테스트 가정(0.05%)과 비교 기록
   - 문제 발견 시 이번 이터레이션은 복구/분석에 전념
3. **업그레이드 1건** (운용 정상일 때): 자체 평가로 최고 가치 항목 선정 → 구현 → 검증
   - 평가 관점: 통계적 정직성 > 리스크 통제 > 비용 절감 > 신규 데이터 엣지 > 예측력
4. **기록**: LOG.md 맨 위에 이터레이션 추가, STATE.md 백로그 갱신
5. **푸시**: `git add -A && git commit && git push` — 모든 변화는 깃허브에
6. **다음 간격**: 업그레이드 작업 중 60초 / 운용 감시만 남으면 30~60분

## 업그레이드 채택 원칙 (전부 이 세션에서 비싸게 배운 것)

- **검증 없이 채택 금지**: 새 아이디어는 9년 walk-forward + 고원 검사 통과해야 운용 반영
- **기각도 성과**: 데이터가 거부하면 기각 사유를 LOG에 남긴다 (ML, 캐리, ADX, 1분봉 등 전례)
- **다중검정 인식**: 같은 데이터에 실험을 쌓을수록 DSR이 나빠진다 — 파라미터 재탐색은 자가학습(월간)에 위임하고, 루프는 구조적 개선에 집중
- **라이브 우선**: 백테스트 개선 < 라이브 정합성/안전장치 개선

## 가드레일 (매 이터레이션 준수)
