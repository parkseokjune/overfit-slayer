## 1. 체결 품질 원장 (Execution Quality Ledger)

**문제**: paper_trades.csv에 체결가만 있고 기준가(신호 시점 가격)·주문유형(maker/taker)·대기시간이 없다.
월간 점검 §2의 "maker 비중·슬리피지" 가 영구 미계측.

**설계**:
- `_execute_with_maker`/`execute_dry_run`이 기록할 때 컬럼 추가:
  `ref_price`(신호 기준가), `fill_price`, `slippage_bps`((fill-ref)/ref×부호), `order_type`(maker/taker/market_fallback), `wait_sec`
- `src/exec_quality.py`: 30일 윈도우 집계 — 평균/최악 슬리피지 bps, maker 체결률, 폴백률
- monthly_check §2가 이 집계를 소비: 기준 `평균 슬리피지 < 10bps(가정 5bps의 2배)`, `maker율 > 50%`
- **수용 기준**: 첫 체결 10건 후 monthly_report에 수치 표시, 기준 위반 시 ⚠

## 2. 하트비트 + 데드맨 스위치

**문제**: 러너 프로세스가 죽으면 (전원/크래시/OOM) 포지션이 방치된다. 현재 유일한 감지자는 사람.

**설계**:
- 러너가 매 틱 `results/heartbeat.txt`에 ts 기록 (1줄, 원자적)
- `scripts/watchdog.sh` (별도 프로세스/cron): 하트비트가 5분 이상 정체 시
  ① 알림(레이어 5) ② `--restart` 옵션 시 러너 재기동 ③ 연속 3회 실패 시 "수동 개입 필요" 마킹
- 데드맨 원칙: 감시자는 러너와 **다른 프로세스**여야 한다 (같이 죽으면 무의미)
  — VPS: systemd `WatchdogSec=300` 활용 (이미 Restart=always 있음 → 하트비트 연동만 추가)
- **수용 기준**: 러너 kill 후 5분 내 자동 재기동 + 이벤트가 로그·알림에 남음

## 3. 캔들 마감 정렬 집행

**문제**: 신호 사이클이 15분 그리드라 일봉 마감(UTC 00:00) 후 평균 7.5분, 최대 15분 늦게 집행.
백테스트는 종가 체결 가정 — 마감 직후 변동이 크면 추적 오차.

**설계**:
- 러너 신호 스케줄을 "다음 마감 시각 + 30초"로 동적 계산 (그 외 시간은 기존 15분 유지 — 스탑 평가용)
- 마감 직후 1회는 `urgent=False` maker 우선 유지하되 대기 60초로 단축
- 측정 먼저: 체결 원장(#1)의 `ref_price`(전일 종가) vs fill로 현재 추적 오차를 30일 실측 → **5bps 초과일 때만** 본 변경 적용 (측정 없이 최적화 금지 원칙)

## 4. 설정 거버넌스

**문제**: self_learn이 config.yaml을 직접 덮어씀. 서버에서 돌면 git 이력과 괴리, 롤백 불가.

**설계**:
- 교체 발생 시 `results/config_snapshots/config_YYYYMMDD_HHMM.yaml` 저장 (변경 전 버전)
- self_learn_history.csv에 `config_snapshot` 컬럼 추가 (롤백 포인터)
- 롤백 명령: `python -m src.self_learn --rollback <snapshot>` (검증: pytest 통과 후 적용)
- experiments/self_learn 기록에 `git_rev` 컬럼 (재현성)

## 5. 알림 추상화 (텔레그램 준비)

**문제**: 경고가 ALERT.txt·로그에만 남음. 텔레그램은 사용자 보류 중이나 결합도를 미리 낮춰야 함.

**설계**:
- `src/notify.py`: `notify(level, msg)` 단일 진입점. 채널 = 환경변수로 활성화
  (기본: 파일+로그 / `TELEGRAM_BOT_TOKEN` 설정 시 텔레그램 / 추후 디스코드 등 1함수 추가)
- 호출 지점: 킬스위치·비상밴드·recon 불일치·드리프트 WARNING+·러너 데드맨·첫 체결
- **수용 기준**: 토큰 넣는 순간 코드 수정 0줄로 푸시 알림 동작

## 6. 복구 모드 (Reconcile Recovery)

**문제**: 포지션 불일치 시 진입 차단까지만. 해소 절차가 수동·비정형.

**설계**:
- `python -m src.recover` 대화형: ① 현황 출력(장부 vs 거래소) ② 선택지 제시
  [A] 거래소를 진실로 — 장부를 거래소에 맞춰 재구성 / [B] 전량 청산 후 플랫 재시작(새 epoch)
  [C] 무시(위험 — 사유 기록 강제)
- 어떤 선택이든 epoch 이벤트로 기록 + recon_block 해제는 이 도구만 가능 (수동 json 편집 금지)
